1. Git介绍

开源、分布式、直接记录快照、专针对数据

1.1 仓库概念

  1. 工作区：workspace，电脑能看到的目录，写代码的地方，通过git init创建的代码库，但不包括 .git文件(版本库)
  2. 暂存区/索引：Index/Stage，保存暂时信息，通过git add操作存入的地方(.git/index)，
  3. 本地库仓库：Repository，本地保存每个版本记录，本地文件夹 + Git 历史记录
  4. 版本库：工作区中的目录(.git，版本库=暂存区+本地库)
  5. 远程库：Remote，远程保存每个记录，托管在服务器上的代码库（如 GitHub/GitLab）

1.2 文件的三种状态

  1. 已修改（modified）：没进行add和commit操作，在工作区
  2. 已暂存（staged）：进行add操作，但没commit操作，保存到暂存区
  3. 已提交（committed）：进行add操作和commit操作，保存到本地库

1.3 配置

1. 配置文件

Git自带git config的工具控制Git外观和行为的配置变量（存储在三个不同的位置）

  1. /etc/gitconfig 文件：包含系统上每个用户及他们仓库的通用配置。 带有--system 选项时，会从此文件读写配置变量。
  2. ~/.gitconfig 或~/.config/git/config文件：只针对当前用户。 --global选项让 Git 读写此文件。
  3. .git/config：当前仓库的config 文件，针对该仓库。

每一个级别覆盖上一级别的配置，即 .git/config 的配置变量会覆盖 /etc/gitconfig 中的配置变量。

2.5 分支管理

版本控制中使用多分支同时推进多个任务，每个分支代表一条独立的开开发线

每个分支并行开发互不影响，对一个分支的增删改查不影响另一个分支，提高开发效率

3. 远程库操作

若是想和其他开发人员合作，需要将数据放到共同的服务器上

GitHub项目的三个操作

  1. Watch/Unwatch:跟踪项目，每当项目发生变动时自己都会收到站内通知，或邮件通知
  2. Star：收藏、点赞，在个人信息列表的your stars目录里收藏着所有Star过的项目
  3. Fork：克隆、分支：相当于对原项目创建分支(包括历史记录)，自己克隆到本地修改后上传，可发起pull request给原项目作者，然后等待他的merge通知

3.1 GitHub配置

由于本地git仓库和远程仓库是通过SSH加密传输，所以需要配置验证信息

$ssh-keygen -t rsa -C “``123@abc.com``”输入邮箱生成SSH Key

  1. 进入.ssh文件下，复制全部id_rsa.pub里面的值
  2. 配置：在Github上，Account >> Settings >> SSH and GPG keys >> New SSH key
  3. 添加：Tittle随便填，复制全部内容进去，保存

$ssh -T ``git@github.com输入命令并键入yes，进行连接验证

4. 团队合作

4.1 团队内部合作

  1. 团队1创建项目，保存origin地址
  2. 团队2根据origin地址进行fork操作，然后克隆到本地仓库进行修改
  3. 修改好后从本地库上传到远程库，并发送pull request请求
  4. 团队1收到pull request通知，并决定是否接受merge
  5. 团队2等待merge通知

4.2 跨团队合作

  1. 团队 A（仓库维护方）
    1. 在 GitHub 新建公开或私有仓库，记录仓库 URL。
    2. 设定主分支保护、Issue/PR 模板、CI 规则。
    3. 定期查看 Pull Request，进行代码评审。
    4. 评审通过后执行合并，并在必要时打版本标签。
    5. 通知贡献方合并结果，关闭相关 Issue。
  2. 团队 B（功能贡献方）
    1. 在 GitHub 将团队 A 的仓库 Fork 到本组织或个人名下。
    2. 把 Fork 后的仓库克隆到本地开发环境。
    3. 创建功能分支，完成需求开发、测试与本地提交。
    4. 将功能分支推送到自己的远程仓库。
    5. 在 GitHub 发起 Pull Request，描述改动并关联 Issue。
    6. 根据团队 A 的评审意见迭代修改，直至合并完成。
    7. 合并后同步最新主分支到本地和远程，清理已完成功能分支。
  3. 系统管理员 / DevOps（可选支撑角色）
    1. 为仓库配置持续集成/持续部署（CI/CD）流程。
    2. 监控主分支健康状态，确保合并前通过全部检查。
    3. 在发布节点创建 Release 分支或 Tag，触发自动部署。
